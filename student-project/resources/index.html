1. Glossary

1.2 Runtime requirements

A runtime requirement is the environment a tool needs to run or build a site. In design education this framing helps students choose workflows that match their project scope. Some tools are instantly usable in the browser; others rely on a local development stack to compile, bundle, or serve content.

1.3 Requires Node.js

Node-based tools use a package manager and scripts to automate tasks such as bundling, transforming CSS, and running development servers. This is the dominant contemporary workflow and valuable for learning industry practice, though it introduces configuration and version management.

1.4 No Node.js

These libraries and frameworks work from a CDN or simple download. They are effective in first-year courses or fast prototypes because they minimise setup and foreground visual exploration over tooling.

1.5 Requires Ruby

Ruby tools such as Jekyll are historically aligned with GitHub Pages. They suit course blogs and documentation sites, offering accessible templating with limited ecosystem breadth compared to Node.

1.6 Standalone binary

Standalone generators like Hugo ship as a single executable. They are portable, extremely fast, and stable across lab machines—useful for large documentation or portfolio sites without dependency drift.

1.7 SaaS

Cloud services provide hosted functionality (content, media, deployment) through a dashboard or API. They are convenient for studio courses and collaborative projects, but introduce platform dependency and ongoing costs.

TailwindCSS — Analysis

Interface quality. Tailwind is not a GUI but its documentation acts as an interface, specially for CSS which is easy to understand, acquire and apply to other contexts.

Discoverability. Class names are systematic and predictable; the docs include copy-ready codes and links, which at the end of the day shortens the quantity of errors.

Functional depth and integration. Requires Node for best use (JIT and PostCSS). It integrates with React and static generators. The main risk is over-reliance on utility if it has not been planned correctly.

Strengths. Design system thinking, rapid iteration, easy to undertand Weaknesses. Build-time dependency; semantic readability may suffer without clear component structure.

Recommended for portfolio programming or design

  1. Responsive project card with tokens (spacing, color scale).
  2. Hero section using gradients and grid layout.
  3. Accessible modal component with focus traps (Tailwind + JS).

Bootstrap — Analysis

Interface quality. The documentation is highly structured with component pages, live examples, and code next to visuals. Quick use of functionals layouts

Discoverability and variety. Patterns are easy to find; the utility API reduces the need for custom CSS.

Functional depth and integration. Works via CDN, Content Delivery Network ; deeper theming requires Sass or build tools. Component JavaScript is minimal and accessible.

Strengths. Fast, predictable, well-tested. Weaknesses. Repetetive viuals, needs to be custumized; heavier CSS footprint than utility-first approaches.

Recommended for portfolio programming or design

  1. Responsive navbar with collapse and active states.
  2. Card grid for project thumbnails using the grid system.
  3. Modal contact form with validation helpers.

link

Hugo — Analysis

Interface quality. Hugo’s CLI,command line interface, is concise; theme demos and partials serve as the practical interface./p>

Discoverability. Documentation includes examples and a theme directory; shortcodes are well covered.

Functional depth and integration. Instantly builds codes. Integrates with Markdown, data files, and external APIs at build time. Requires learning Go-templating idioms.(!!!)

Strengths. Speed, stability, portability. Weaknesses. Smaller ecosystem of plugins; advanced interactivity requires client-side JS.

Recommended for portfolio programming or design.

  1. List templates for projects with taxonomies.
  2. Shortcode for responsive images and captions.
  3. Partial for SEO and social card metadata.

link

Jekyll — Analysis

Interface quality. Clear directory conventions and Liquid templating provide a gentle learning curve, suitable for reflective journals and blogs. Adaptable to portfolio.

Discoverability.GitHub Pages integration removes deployment friction, easy to adapt to my work.

Functional depth and integration. Plugin and other assets: scarce; build times are slower than Hugo. Ideal for Markdown-centred content rather than heavy interaction.

Strengths. Simplicity and hosting convenience. Weaknesses. Limited extensibility

Recommended portfolio code

  1. Layout and include structure (header, footer, SEO).
  2. Post list with excerpts and tags.
  3. Data-driven project pages using YAML.

link

React — Analysis

Interface quality. React is a library of codes, rather than templates. Easy to search in.

Discoverability. Extensive documentation and community help/input; integration examples with routing and data-fetching

Functional depth and integration. Excels at interactive exhibits and micro-apps within a portfolio. Requires Node and a bundler; complexity rises with ecosystem choices.

Strengths. Composability and reusability. Weaknesses. Too many tools can make projects confusing, and performance problems can appear if they aren’t tested carefully. (to really take into consideration)

Recommended portfolio code

  1. Project card component with props and accessibility.
  2. Lightbox gallery with keyboard navigation.
  3. Interactive data viz using a small chart library.

GSAP — Analysis

Interface quality. GSAP is based on conceptual interface that clarifies motion sequencing; docs feature runnable demos but not full on codes.

Discoverability. The examples directory and articles make it easy to access and undertand what the code is for.

Functional depth and integration. Works from CDN; integrates with scroll libraries and frameworks. Some codes need polishing regarding their hierarchy./p>

Strengths. Precision, performance, community. Weaknesses. Complexity for beginners; proprietary plugins for some features. Which are hard to come by.

Recommended portfolio code

  1. Hero sequence with staggered reveals.
  2. Scroll-driven storytelling timeline.
  3. Micro-interactions for hover and focus states.

Material UI — Analysis

Interface quality.The documentation is really good — it shows clear examples, explains how each part works, and even has interactive previews where you can try things out on your own-to make sure they work

Discoverability. Clear navigation and example gallery.

Functional depth and integration. Requires React and Node; strong accessibility defaults. Styling decisions can become repetitive, need to custumize.

Strengths. Consistency, accessibility, and scalability. Weaknesses. Material UI has a very recognisable visual style. Also, if the project isn’t well-optimised, it can include unnecessary code and make the website heavier or slower to load

Recommended portfolio code

  1. Dashboard with navigation drawer and app bar.
  2. Data table with sorting and pagination.
  3. Theme switcher demonstrating tokens.

Flowbite — Analysis

Interface quality. The documentation has short copy-paste code examples that make it quick to build common website sections like landing pages or marketing layouts.

Discoverability. Components are grouped by purpose; examples are concise and easy to adapt.

Functional depth and integration. Pairs best with Tailwind projects. Over-reliance can be dangerous (like I've said the last times haha).

Strengths. Velocity and coherence. Weaknesses. Risk of generic aesthetic; limited beyond Tailwind.

Recommended portfolio code

  1. Pricing section with responsive cards.
  2. Accessible navigation with dropdowns.
  3. Testimonials slider using utility classes.

link

Hygraph — Analysis

Interface quality. The dashboard makes it easy to create content types, manage user access, and edit entries. It’s clearly organised, and the built-in GraphQL explorer helps you understand how to work with data.

Discoverability. Starters and quickstarts cover common stacks; schema examples translate well to portfolio content types.

Functional depth and integration. SaaS reliability with webhooks and API access. Introduces external dependency and potential cost considerations after graduation.Which is not an option since I have other possibilities.

Strengths. Structured content at scale. Weaknesses. Vendor lock-in; requires network access.

Recommended portfolio code

  1. GraphQL query to list projects by category.
  2. Static build integration example (SSG).
  3. Webhook trigger for rebuilds on publish.

link

ImageKit.io — Analysis

Interface quality. The dashboard clearly shows where images come from, how they’re processed, and how they’re stored. Its URL-based system is easy to remember and simple to test in the browser.

Discoverability. Guides show responsive image patterns for multiple frameworks; code samples are concise.

Functional depth and integration. CDN delivery, automatic format negotiation, and transformations improve portfolio performance. Reliance on a third-party service must be acknowledged !!!

Strengths. Performance and global delivery. Weaknesses. Platform dependency and quotas.

Recommended portfolio code

  1. Srcset pattern for responsive images.
  2. On-the-fly transformation URLs.
  3. Low-quality image placeholders for perceived speed.

4. Critical Reflection

This research helped me understand how different tools and classifications-the reason why to the glossary. Node-based tools like Tailwind, React, and Material UI have shown me how web stacks work — they make it easier to build reusable design systems, but they also require more setup and attention. For my portfolio projects, these are great once I already become familiar with how they work and exactly where I want to use the code. But I need to be aware of hidden costs.

On the other hand, tools like Bootstrap and GSAP are much quicker to start with. They’re ideal for testing layouts or animations when I want to focus on visual language rather than configuration. Which as a graphic designer is vital to understand if the code is harder to descpher. Hugo and Jekyll have fast and predictable builds — something really useful for submitting work that’s downloadable and consistent across devices. But contradictory To what I really give as a graphic designer. So is really important for me to custumize in an easy manner. Meanwhile, Hygraph and ImageKit opened up possibilities for managing dynamic content and optimised visuals online, which I could apply to my portfolio.

Overall, once all the tools have been analyzed, it is important to note the complexity of my portfolio and my need to design, accomodate and learn. Which is mostly based now in the use of nodes and a mixt of GSAP since the layout was really fun and enticing. Which is always a positive when designing.
